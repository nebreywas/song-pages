# Security Model and Currently Completed Actions

**Status:** Canonical · **Index:** [README.md](./README.md)

This document summarizes the Song Pages desktop application security architecture and what has been implemented in the proof-of-concept codebase as of this writing.

For full detail on guest webview behavior, see [guest-rendering-security.md](./guest-rendering-security.md). For manifest contracts, see [manifest-schemas.md](./manifest-schemas.md).

---

## Core principle

**Artists are trusted to create interesting presentation. They are not trusted with application privileges.**

Remote Song Page HTML/CSS/JS is always treated as untrusted web content, even when generated by the Song Pages compiler. The Electron app maintains a hard boundary between:

| Trusted (application) | Untrusted (guest) |
|----------------------|-------------------|
| React shell, preload bridge, main process | Remote Song Pages loaded in webview |
| SQLite, IPC, filesystem, ffmpeg compile | Artist-hosted HTML, JS, CSS |
| App HLS player | Site footer player (browser mode only) |

---

## Security model

### 1. Process and API isolation

- **Main window renderer**: `contextIsolation: true`, `nodeIntegration: false`, preload exposes audited `window.app` API only.
- **Guest webview**: separate hardened session partition (`persist:songpages-guest`), **no preload**, **no IPC**, **no Node**, **`sandbox: true`**, **`contextIsolation: true`**.

Artists cannot call Electron APIs, access SQLite, read the filesystem, or escape into the app shell through normal web APIs.

### 2. Guest session hardening

Implemented in `electron/listener/guestSession.js`, configured at app startup:

- All permission requests denied (camera, microphone, location, notifications, etc.)
- Downloads from guest content blocked (`will-download` prevented)

### 3. Navigation and window policy

Implemented in `electron/listener/guestSecurity.js`, bound per guest via IPC `listener:bindSongPageGuest`:

- **Same song page only**: navigation allowed when origin and pathname match the loaded canonical page (hash/query changes OK)
- **Everything else**: navigation prevented; URL opened in the **system default browser**
- **`window.open` / popups**: denied; target URL opened externally
- Stream links (Spotify, YouTube, etc.) naturally open outside the app

### 4. App presentation mode (not security, but reduces attack surface)

Listener Mode loads song pages with `?songpagesApp=1`.

- **Electron does not inject CSS or JavaScript** into guest documents
- Compiled templates recognize the query param and add class `songpages-app-client`
- Browser-only chrome hidden via template CSS (`data-songpages-client-chrome`): footer player, home button, inline play buttons
- `site-player.js` exits early in app mode (no duplicate HLS player in guest)
- Legacy `songpagesEmbed=1` still recognized during transition

Constants: `shared/appClient.ts`

### 5. Content Security Policy (compiler-baked)

Every compiled HTML page receives a CSP meta tag via `compiler/staticSiteBuild.ts` → `injectContentSecurityPolicy()`:

- `default-src`, `script-src`, `style-src`, `media-src`, `connect-src` limited to `'self'`
- `form-action 'none'` — no form submission
- `frame-src 'none'` — no embedded frames
- `object-src 'none'`

CSP is defense-in-depth alongside Electron sandboxing. It limits third-party resource loading if a site is modified after compile.

Definition: `shared/siteCsp.ts`

### 6. Main window vs guest (intentional split)

The **main application window** may use relaxed settings required for PoC functionality:

- `sandbox: false` on parent (required for `<webview>` guest rendering)
- `webSecurity: false` on parent (allows app HLS player to fetch `.m3u8`/`.ts` from CDNs without CORS headers)

These relaxations apply **only to the trusted app shell**, not to guest webviews. Guest pages use sandboxed partition + strict webpreferences.

### 7. Projection windows (Visualizer + VC Mode)

Two additional **trusted** renderer windows support streaming and second-display use cases:

| Window | File | Purpose |
|--------|------|---------|
| Visualizer | `electron/visualizerWindow.js` | External audio-reactive projection |
| VC Mode | `electron/vcWindow.js` | Shareable listening-party surface + **mirrored HLS audio** for window capture |

Both use the same preload as the main window (`electron/preload.js`) with `contextIsolation: true`, `nodeIntegration: false`, and **`webSecurity: false`** (CDN HLS without CORS). VC window additionally sets `autoplayPolicy: 'no-user-gesture-required'` so mirrored playback can start without user gesture in the capture target.

These windows do **not** load artist song page guests. They render app-controlled React surfaces only. See [vc-mode-architecture.md](./vc-mode-architecture.md) and [visualizer-architecture.md](./visualizer-architecture.md).

**Future hardening:** Narrow preload API surface for projection windows; media proxy instead of global `webSecurity: false`.

### 8. Host content media (local imports)

Host graphics and videos are imported via IPC (`hostContent:pickAndImportMedia`) into `{userData}/host-content/media/`. Validation (format, size, dimensions for graphics) runs in main process (`electron/hostContent.js`). Catalog metadata persists in SQLite key `vc.hostContent`. This is trusted local user content, not remote artist HTML.

See [archive/specs/Host-content-design.md](./archive/specs/Host-content-design.md) and [settings-and-persistence.md](./settings-and-persistence.md).

## Not yet implemented (future work)

- Replace main-window `webSecurity: false` with a safer media proxy for HLS (narrower than disabling web security globally) — applies to main, visualizer, and VC windows
- Narrow preload API exposed to projection windows (currently full `window.app`)
- Signed manifests or manifest authenticity verification
- Allowlist per-artist site roots beyond pathname matching
- User-visible message when external link opens in browser
- Production packaging hardening (notarization, code signing)
- Automated security tests for guest escape attempts
- Remove legacy `songpagesEmbed` support after all sites recompiled

## Operational notes

1. **Recompile + re-upload** artist sites after template/CSP changes for full app-mode chrome hiding and CSP on CDN.
2. Electron guest security binding applies to **all** remote song page URLs immediately after app restart.
3. Remote test site: `https://sawyerhouse-music.b-cdn.net` (requires manifests from new compiler).

## PoC success criteria (reference)

The PoC is successful when:

1. Artist Mode compiles a static 12-song site with three manifest types
2. Site is manually uploaded to static host
3. Listener subscribes by URL, imports catalog to SQLite, plays HLS, shows song page in sandboxed webview
4. App presentation mode hides duplicate browser chrome
5. Guest content cannot access app privileges
6. Library persists across restarts
